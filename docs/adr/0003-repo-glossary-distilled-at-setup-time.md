# ADR-0003: プロジェクト用語集は /setup 時に GitHub リポジトリから蒸留してキャッシュする

| 項目 | 内容 |
|---|---|
| ステータス | Accepted |
| 日付 | 2026-07-12 |
| 決定者 | hinapupil（grill-with-docs セッションで確定） |

## Context

Speech-to-Text はプロジェクト固有名詞を高頻度で誤変換する（実測例: "Proto Pedia" → 「グッドペディア/フロントペディア/ベディア」、"MiyaIF" → 「みやいふ/宮井さん」、"金曜" → 「禁酒」）。Gemini は文脈から補正を試みて注記を残すが、プロジェクト固有の語彙を知らないため推測の域を出ない（[issue #26 のユースケース0](https://github.com/hinapupil/minutes-agent/issues/26#issuecomment-4951208877)）。

プロジェクトオーナーの要望は「静的な用語集ではなく、agent が指定 GitHub リポジトリを読んで**自律的に**用語を学習・注入すること」。実装方式の選択肢:

- **案A（setup 時蒸留 + キャッシュ）**: `/setup repo:<owner/repo>` 実行時に agent がリポジトリ（README・docs/*.md・ファイルツリー・コントリビューター名）を読み、Gemini で「音声認識補正用の用語集」を蒸留して Firestore `guild_settings/{guild_id}` に保存。議事録生成時はキャッシュを注入するだけ
- **案B（生成時にツールで読む）**: ADK agent に fetch_repo ツールを持たせ、議事録生成のたびに agent 自身が必要箇所を読みに行く。「自律的」の純度は最高だが、生成レイテンシ増・失敗面の増加・GitHub API rate limit（無認証 60 req/h）に毎回さらされる
- **案C（生成時に生全文注入）**: 毎回リポジトリ全文をプロンプトに入れる。トークンコスト・レイテンシで非現実的

## Decision

**案A を採用する。** 「agent がリポジトリを自律的に読む」体験は setup 応答（学習した用語集のプレビュー返信）で成立し、議事録生成は決定論的・高速のまま保てる。読む範囲は文書+メタデータ（README・docs 配下 markdown・ファイルツリー・コントリビューター名・リポジトリ説明）に限定し、ソースコード本文は読まない — 固有名詞の補正が目的なら十分で、蒸留入力が数万トークンに収まる。

- 上限: markdown 最大 15 ファイル・合計 300KB、用語は最大 60 語
- 全 HTTP リクエストに識別可能な User-Agent を付与（GitHub API は UA なしを 403 で拒否。Discord/Cloudflare の error 1010 と同じ教訓）
- 未設定ギルド・Firestore 読み込み失敗時は用語集なしで生成続行（用語集は品質向上のオプショナル要素であり、障害点にしない）

## Consequences

**得られるもの:**
- 用語集に載る固有名詞の誤変換を仕組みで潰せるようになり、議事録の実用品質が上がる
- ギルド単位設計（`guild_settings/{guild_id}`）がマルチギルド展開（[#57](https://github.com/hinapupil/minutes-agent/issues/57)）の布石になる
- 生成経路のレイテンシ・信頼性を維持

**失うもの / 引き受けるリスク:**
- 用語集の鮮度は `/setup` 再実行に依存する（リポジトリ更新への自動追随なし）。webhook 連動更新は #26 の将来スコープ
- 会議中に初出の語彙（リポジトリに無い語）は補正できない
- private リポジトリ非対応（無認証取得のため）。GitHub App 認証は #26 で扱う

**進化パス:** 案B（生成時ツール化）へは、蒸留キャッシュを「一次情報への索引」として残したまま fetch ツールを追加する形で移行できる（#26 に記録済み）。
